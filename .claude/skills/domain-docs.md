# 도메인 문서 작성 가이드

`docs/domain/{도메인명}.md` 파일을 작성하거나 업데이트하는 스킬.

## 문서 위치

- 출력 경로: `docs/domain/{도메인명}.md`

## 작성 절차

### 1. 도메인 범위 파악

대상 도메인에 해당하는 소스를 탐색하여 문서에 필요한 정보를 수집한다:

| 정보 | 탐색 위치 |
|------|----------|
| 데이터 모델 | `src/lib/db/schema.ts` — 해당 도메인의 테이블, 컬럼, FK, 인덱스 |
| API 라우트 | `src/app/api/` — 해당 도메인의 라우트 핸들러 |
| 서비스 로직 | `src/lib/services/` — 비즈니스 로직, 특수 처리 규칙 |
| 페이지 | `src/app/(main)/` — UI 페이지, 주요 기능 |
| 훅 | `src/hooks/` — React Query 훅, 상태 관리 훅 |
| 타입 | `src/types/` — 도메인 관련 타입 정의 |

### 2. 문서 구조 (섹션 순서 고정)

```markdown
# {도메인명} 도메인

한 줄 요약. 이 도메인이 무엇을 관리하고, 핵심 특징은 무엇인지.

## 데이터 모델

### {테이블명}

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | UUID (PK) | |
| ... | ... | ... |

**관계:** {부모} 1 : N {자식}

## 페이지

| 경로 | 기능 |
|------|------|
| `/도메인/...` | 기능 설명 |

### {페이지명} 주요 기능 (필요 시)

- **기능명**: 상세 설명

## API

### {기능 그룹}

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | `/api/...` | ... |

## 비즈니스 로직

### {로직명}

핵심 규칙, 처리 흐름, 특수 케이스 설명.

## 다른 도메인과의 연관

| 연관 도메인 | 관계 |
|-------------|------|
| **도메인명** | 연관 설명 |

## 주요 파일

| 파일 | 역할 |
|------|------|
| `src/...` | 역할 설명 |
```

### 3. 작성 원칙

- 코드에서 직접 확인한 사실만 기록한다. 추측하지 않는다.
- 테이블 컬럼의 타입은 스키마 정의 기준으로 작성한다 (예: `varchar(255)`, `numeric(10,2)`, `UUID (FK → 부모테이블)`).
- FK의 `onDelete` 정책을 설명에 명시한다 (예: `CASCADE DELETE`).
- API 엔드포인트는 실제 라우트 파일 구조에서 확인한다.
- 비즈니스 로직은 서비스 파일의 핵심 규칙만 간결하게 정리한다. 구현 상세보다 "무엇을 하는가"에 집중한다.
- 주요 파일 섹션에는 해당 도메인과 직접 관련된 파일만 나열한다.
- 해당 도메인에 없는 섹션은 생략한다 (예: UI가 없는 도메인이면 페이지 섹션 생략).

## 업데이트 시

기능 변경 작업 후 도메인 문서를 업데이트할 때는 변경된 부분만 반영한다:

- 테이블/컬럼 추가·변경·삭제 → 데이터 모델 섹션 업데이트
- API 추가·변경 → API 섹션 업데이트
- 비즈니스 로직 변경 → 비즈니스 로직 섹션 업데이트
- 새 파일 추가 → 주요 파일 섹션에 추가
- 파일 삭제·이동 → 주요 파일 섹션에서 제거·경로 수정
