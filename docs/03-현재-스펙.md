# 현재 스펙

이 문서는 "현재 코드가 실제로 제공하는 계약"을 정리한 문서입니다.

## 1. 화면 스펙

### `/`

- 프로젝트 목록 조회
- 새 프로젝트 생성

### `/boxes`

- 전역 박스 목록 조회
- 박스 추가
- 박스 삭제

### `/projects/:id`

- 프로젝트 대시보드
- 현재는 상품 수 / 출고 수 중심 요약

### `/projects/:id/products`

- 상품 엑셀 업로드
- 상품 일괄 등록
- 등록 상품 목록 조회
- 선택 삭제 / 전체 삭제

### `/projects/:id/outbound`

- 출고 엑셀 업로드
- 저장 결과 및 미매칭 결과 확인

### `/projects/:id/outbound/list`

- 주문 단위 출고 목록 조회
- 주문 단건 패킹 계산
- 전체 패킹 계산 후 요약 이동

### `/projects/:id/packing`

- grouping 옵션 선택
- 전체 패킹 계산
- 패킹 이력 조회
- 결과 엑셀 export

### `/projects/:id/packing/summary`

- 계산 결과를 박스 기준으로 요약해서 확인

## 3. 도메인 스펙

### Project

- 프로젝트 단위의 최상위 스코프
- 상품, 주문, 출고, 패킹 결과가 프로젝트 기준으로 묶임

### Product

- 프로젝트별 상품
- `(project, sku)` 기준으로 사실상 유니크
- `width`, `length`, `height` 보유

### Order

- 프로젝트별 주문
- 외부 주문번호는 `orderId`
- 상태는 `PENDING | PROCESSING | COMPLETED`

### Outbound

- 실제 출고 아이템
- 내부 relation용 `orderId`와 외부 주문번호 보존용 `orderIdentifier`가 공존할 수 있음
- `productId`는 nullable

### Box

- 전역 박스
- 모든 프로젝트가 공유

### PackingResult / PackingResultDetail

- 계산 결과 저장용 엔티티
- 프로젝트 단위 snapshot 성격

## 4. 패킹 스펙

현재 패킹 알고리즘의 핵심 규칙:

- 박스는 부피 오름차순 정렬
- SKU 수량은 낱개 아이템으로 펼쳐서 계산
- 회전 허용은 치수 정렬 비교 방식으로 처리
- 박스 유효 용량은 90% 사용
- 남은 아이템 전체가 한 박스에 들어가면 가장 작은 가능한 박스를 선택
- 아니면 가장 큰 박스를 그리디하게 채움

즉, 현재 구현은 엄밀한 3D bin packing보다는:

- 부피 기반
- 치수 적합성 체크
- 그리디 분할

성격이 강합니다.

## 5. 현재 구현상 제한 사항

### 상품 업로드

- AI 스키마는 `dimensionFormat: separate`를 표현할 수 있음
- 하지만 실제 변환 로직은 결합형 dimensions 문자열 처리 위주

### 출고 업로드

- AI는 `recipientName`, `address`까지 추정 가능
- 하지만 direct upload 저장 경로에서는 이 정보가 Order에 충분히 반영되지 않음

### 주문 ID 처리

- Outbound는 내부 relation id를 `orderId`에 저장
- 외부 원본 주문번호는 `orderIdentifier`에 따로 저장
- UI에서는 주로 `orderIdentifier`를 우선 사용

### 패킹 결과 저장

- 프로젝트 전체 계산 시 기존 결과를 삭제하고 다시 생성
- 누적 히스토리형 저장 구조는 아님

### 3D 계산

- `PackingResult3D` 타입과 `calculate-order` 이름이 존재
- 하지만 실제 placement 좌표 배열은 비어 있음
