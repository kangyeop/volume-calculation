# 프로젝트 개요

## 이 프로젝트가 하는 일

이 저장소는 프로젝트 단위로 상품 데이터를 관리하고, 출고 엑셀을 업로드한 뒤, 등록된 박스를 기준으로 패킹/CBM 계산을 수행하는 WMS 성격의 모노레포입니다.

핵심 사용자 흐름은 아래 3개입니다.

1. 프로젝트 생성 후 상품 등록
2. 출고 엑셀 업로드 및 상품 매칭
3. 박스 기준 패킹 계산과 결과 확인

## 현재 사용자 입장에서 보는 흐름

### 1. 프로젝트 생성

- `/`에서 프로젝트를 생성
- 생성 후 `/projects/:id`로 이동

### 2. 상품 등록

- `/projects/:id/products`에서 상품 엑셀 업로드
- 서버가 파일을 파싱
- AI가 `sku`, `name`, `dimensions` 컬럼을 추정
- 확인 단계에서 상품을 일괄 저장

### 3. 출고 업로드

- `/projects/:id/outbound`에서 출고 엑셀 업로드
- 서버가 파일을 파싱
- AI가 출고 관련 컬럼을 추정
- 현재 구현은 direct upload 경로를 사용해 바로 저장
- 이때 기존 프로젝트 출고 데이터는 먼저 비워지고 새 데이터로 대체됨

### 4. 출고 검토와 계산

- `/projects/:id/outbound/list`에서 주문별 출고 목록 확인
- 주문 단건 계산 가능
- 전체 계산 후 `/projects/:id/packing/summary`에서 요약 확인 가능

### 5. 박스 관리

- `/boxes`에서 박스 정의 관리
- 박스는 프로젝트별이 아니라 전역으로 공유됨

## 현재 시스템의 범위

현재 코드 기준으로 구현된 범위:

- 프로젝트 CRUD
- 상품 등록/삭제
- 출고 등록/삭제
- AI 기반 컬럼 매핑
- 상품명/SKU 기반 상품 매칭
- 패킹 추천 계산
- 패킹 결과 엑셀 내보내기

현재 코드 기준으로 아직 없는 것:

- 인증/권한
- 재고 차감
- 멀티 창고 개념
- 실제 3D 좌표 기반 적재 최적화

## 빠르게 이해해야 할 포인트

- `apps/server`가 실제 업무 로직의 중심입니다.
- `apps/web`은 이 API를 그대로 호출하는 UI입니다.
- `packages/types`가 프론트/백엔드 공통 계약입니다.
- 업로드와 패킹은 이름상 풍부해 보여도, 실제 구현은 "현재 동작" 기준으로 봐야 합니다.

특히 아래는 처음부터 알고 보는 편이 좋습니다.

- 상품 업로드는 실제로는 결합형 dimensions 문자열 처리 위주입니다.
- 출고 direct upload는 수령인/주소를 Order에 완전히 반영하지 않습니다.
- `calculate-order`는 이름과 달리 실제 placement 좌표 계산은 하지 않습니다.
