# 패킹 알고리즘

위치: `apps/server/src/modules/packing/algorithms/packing.algorithm.ts`

## 알고리즘 개요

부피 기반 그리디 패킹 알고리즘입니다. 실제 3D 좌표를 계산하지 않고, 부피와 치수 적합성만으로 박스 배정을 결정합니다.

## 동작 순서

### 1단계: 전처리

```text
박스 목록 → 부피 오름차순 정렬
SKU 목록 → 수량만큼 낱개 아이템으로 펼침

예: SKU-A (수량 3) → [SKU-A, SKU-A, SKU-A]
```

### 2단계: 치수 적합성 검사 (canFit)

아이템이 박스에 물리적으로 들어갈 수 있는지 확인합니다.

```text
아이템 치수 [w, l, h] → 오름차순 정렬 → [min, mid, max]
박스 치수  [w, l, h] → 오름차순 정렬 → [min, mid, max]

아이템.min ≤ 박스.min AND
아이템.mid ≤ 박스.mid AND
아이템.max ≤ 박스.max → 적합
```

이 방식은 회전을 암묵적으로 허용합니다 (치수를 정렬해서 비교하므로).

### 3단계: 유효 용량 계산

```text
박스 유효 용량 = 박스 부피 × 0.9 (90%)
```

10%는 여유 공간으로 남겨둡니다.

### 4단계: 그리디 패킹

```text
남은 아이템이 있는 동안:
  ① 전체 아이템이 하나의 박스에 들어가는가?
     → 가능한 가장 작은 박스를 선택
     → 모든 아이템을 넣고 종료

  ② 아니라면:
     → 가장 큰 박스를 선택
     → 부피 기준으로 가능한 만큼 아이템을 채움
     → 채운 아이템을 남은 목록에서 제거
     → ①로 돌아감
```

### 5단계: 결과 계산

```text
각 박스별:
  usedVolume = 담긴 아이템 부피 합
  efficiency = (usedVolume / boxVolume) × 100

전체:
  totalCBM = Σ (박스 부피 / 1,000,000)   ← cm³ → m³ 변환
  totalEfficiency = 평균 efficiency
```

## 그룹핑 옵션

패킹 계산 전에 출고 아이템을 그룹으로 나눕니다.

| 옵션 | 그룹핑 기준 | 예시 groupLabel |
|------|------------|----------------|
| ORDER | 주문번호(orderId) | "ORD-001" |
| RECIPIENT | 수령인(recipientName) | "홍길동" |
| ORDER_RECIPIENT | 주문번호 + 수령인 | "ORD-001_홍길동" |

각 그룹은 독립적으로 패킹이 계산됩니다 (그룹 간 박스 공유 없음).

## 제한사항

- 부피 기반 계산만 수행, 실제 3D 좌표 배치 없음
- `PackingResult3D`, `Placement3D` 타입은 존재하지만 좌표 데이터는 비어 있음
- 그리디 알고리즘이므로 최적해를 보장하지 않음
- 아이템 무게를 고려하지 않음
- 같은 박스 타입 내에서 아이템 순서는 부피 기준

## 결과 저장 구조

계산이 완료되면 두 테이블에 저장됩니다:

**PackingResult** (그룹별 요약)
- 그룹당 1개 레코드
- boxName, packedCount, efficiency, totalCBM, groupLabel

**PackingResultDetail** (아이템별 상세)
- 아이템당 1개 레코드
- orderId, sku, productName, quantity, boxName, boxNumber, boxCBM, efficiency
- unpacked=true인 경우 unpackedReason 포함

재계산 시 기존 결과는 삭제 후 새로 생성됩니다 (누적 히스토리 아님).
